Introduction

This program provides a flexible and scalable way to share and synchronize data between multiple Siemens S7-1200 PLCs. Instead of the traditional Siemens PUT/GET method where PLCs communicate directly, this server acts as a centralized intermediary: reading data from each PLC and writing it back to all PLCs in a designated synchronization data block. This allows every PLC to have a real-time snapshot of each other’s data.

In addition to its core PLC data synchronization functionality, the server supports MQTT messaging for real-time data exchange. With MQTT integration, the server publishes combined PLC data, a server heartbeat, and external MQTT updates to designated topics. It also subscribes to MQTT topics to receive commands or data updates, enabling seamless integration with external systems and modern IoT platforms.

Furthermore, the web interface lets you change variables easily, making it straightforward to adjust parameters and test behavior without modifying the PLC’s code directly.

Key Features

  • Multi-PLC data sharing and synchronization.
  • HTTPS JSON interface for monitoring combined PLC data and read/write counters.
  • Flexible configuration through a single JSON file.
  • Automated calculation of synchronization offsets, reducing manual configuration.
  • Rotating log files for troubleshooting and monitoring.
  • Web-based GUI for visualization, configuration changes, and control.
  • Field Manager (/fields): add or remove shared fields across all PLCs from one place. It recomputes every offset (each PLC's share_this_data and the combined sync DB) and generates importable TIA .db external-source files, so the three edit sites stay consistent.
  • Pause/Resume Sync (on the Setup page): temporarily suspends only the cross-PLC sync writes while you change DB layouts, without stopping the process.
  • A built-in manual for reference.
  • Real-time MQTT messaging integration for data publishing and subscription.

Requirements

  • Node.js (v12 or higher recommended)
  • Siemens S7-1200 PLCs with PUT/GET communication enabled
  • Data Blocks (DBs) in the PLCs must have optimized block access disabled to allow absolute addressing.
  • PM2 or another Node.js process manager (optional but recommended).

How It Works

The server periodically reads data from all configured PLCs. It then writes this combined dataset back into each PLC’s designated synchronization DB, ensuring that every PLC can read the same data from all peers. A server heartbeat variable is also updated to confirm that the data is fresh and communication is ongoing.

Installation

  1. Clone the repository or copy the files to your desired location.
  2. Run npm install to install required Node.js modules.
  3. Edit config.json to match your PLC setup (see details below), or use the Field Manager to add/remove shared fields safely.
  4. Start the server with node sws3.js.
  5. This deployment runs under PM2 as plc-sync: pm2 start sws3.js --name plc-sync. Restart with pm2 restart plc-sync (or via the Restart button on the Setup page).

HTTP API & Programming Examples

Important Note:

Since the server is now running over HTTPS with Basic Authentication enabled, all API calls must use https:// URLs and include valid credentials. For example, using curl you can include the -u username:password option.

  • GET /data
    Returns a JSON object containing the server's heartbeat, combined PLC data, and counters.
    Example using curl: 
    curl -k -u admin:supersecret https://your-server-ip:port/data
  • GET /status
    Returns a JSON object describing the status of the server and each PLC.
    Example: 
    curl -k -u admin:supersecret https://your-server-ip:port/status
  • GET /manual
    Returns this HTML manual.
  • GET /gui
    Returns the graphical web interface for managing PLC data and configuration.
    Example: 
    curl -k -u admin:supersecret https://your-server-ip:port/gui
  • GET /plc/<plcName>
    Returns the latest data from the specified PLC in JSON format.
    Example: 
    curl -k -u admin:supersecret https://your-server-ip:port/plc/PLC_MAIN
  • POST /plc/<plcName>
    Writes data to the specified PLC. The request body should be JSON containing variable names and values.
    Example: 
    
curl -k -X POST https://your-server-ip:port/plc/PLC_MAIN \
     -u admin:supersecret \
     -H "Content-Type: application/json" \
     -d '{"machine_running": true, "current_speed": 1200}'
  • GET /setup
    Returns the setup page for configuration editing, credential updating, log viewing, and server restart.
  • POST /config
    Replaces the current configuration with a new JSON object (body must contain a plcs array). The current config.json is first copied to config.json.bak; if that backup fails the write is aborted. The new config takes effect on the next restart. This is what the Setup editor and the Field Manager's Replace config.json button call.
    Example: 
    
curl -k -X POST https://your-server-ip:port/config \
     -u admin:supersecret \
     -H "Content-Type: application/json" \
     -d '{ "operationInterval": 1000, "plcs": [ ... ] }'
  • GET /fields
    Returns the Field Manager page (add/remove shared fields, generate importable .db files).
  • POST /fields/preview
    Generates the new config + .db artifacts for an add/remove operation, without writing anything. Body: { "op": "add"|"remove", "plc": "<name>", ... } — add takes a single name+type or a fields array; remove takes a single name. Returns { newConfig, shareDb, syncedDb, preview }.
  • POST /sync/pause / POST /sync/resume / GET /sync/status
    Pause or resume only the cross-PLC sync write action (reads, status and the web server keep running); query the current state. The flag is in-memory, so a restart always resumes syncing.
    Example: 
    curl -k -X POST https://your-server-ip:port/sync/pause -u admin:supersecret
curl -k https://your-server-ip:port/sync/status -u admin:supersecret   # {"paused":true}
  • POST /update-credentials
    Updates the login credentials. The request body must include a new username and password.
    Example: 
    
curl -k -X POST https://your-server-ip:port/update-credentials \
     -u admin:supersecret \
     -H "Content-Type: application/json" \
     -d '{"username": "newUser", "password": "newPass"}'
  • POST /restart
    Instructs the server to exit gracefully (which will trigger a restart if managed by a process manager).
    Example: 
    curl -k -X POST https://your-server-ip:port/restart -u admin:supersecret
  • GET /logs
    Returns an array of available log filenames.
    Example: 
    curl -k -u admin:supersecret https://your-server-ip:port/logs
  • GET /logs/<logFile>
    Returns the last 100 lines of the specified log file as plaintext.
    Example: 
    curl -k -u admin:supersecret https://your-server-ip:port/logs/plc-app-2025-01-31.log
  • GET /logout
    Logs out the current user by forcing a re-prompt of credentials.
    Example: 
    curl -k -u admin:supersecret https://your-server-ip:port/logout

Data Synchronization and Server Heartbeat

The server writes a heartbeat variable to the sync DB to verify data freshness. PLCs monitor this variable to detect communication issues. Additionally, each PLC's reachable and alive status is tracked.

Example Pattern: a _req / _ack / _state Handshake

The sync server attaches no meaning to field names or values — it simply transports each PLC's shared bytes to every other PLC. You are therefore free to structure the shared data however you like. The pattern below is just one way the PLC programs at this site choose to coordinate over the shared data; it is a convention in the PLC logic, not something the sync layer requires or enforces.

In this pattern, fields are named with three suffixes. Each field is written by the PLC in whose share_this_data it is configured, and read by every other PLC via the sync DB (DB200):

  • _req / _request — the desired state or value, written by the requesting PLC. It may carry a setpoint (e.g. pump_speed_request [INT], heating_temperature_request [REAL]) or be a simple on/off BOOL.
  • _ack (BOOL) — also written by the requesting PLC. false = a request is open / in progress; true = the executor has reached the requested state.
  • _state / _stat — the actual current state or feedback, written by the executing PLC; the value the requester compares against.

The handshake (as implemented in the PLC logic)

  1. The requesting PLC writes the desired state into _req and sets _ack = false — this opens the request.
  2. The executing PLC looks for a requester whose _ack is false and drives its output to that _req state, reflecting it in its own _state.
  3. The requesting PLC compares the executor's _state against _req; once they match it sets _ack = true, closing the handshake.

That is why, in this pattern, the requesting PLC owns both _req and _ack while the executing PLC owns _state. It is level-based and tolerant of the cyclic sync latency: while _ack stays false the executor keeps driving toward _req, and the requester only confirms once the real _state actually matches.

Multiple requesters each use their own _req/_ack pair, so independent requests run as parallel handshakes. For a shared output the executor services the most recent open request (last come, last served); this stays stable as long as a satisfied requester (_ack = true) does not re-open when a later request moves the shared _state.

Example. To start a pump on PLC_B, PLC_A sets pump_a_req = true and pump_a_ack = false. PLC_B sees the open request, runs the pump and publishes pump_a_state = true. PLC_A sees pump_a_state match its request and sets pump_a_ack = true. The reverse direction works the same way (e.g. PLC_A requests pump_b_motor from PLC_B, which publishes pump_b_motor_state), and a request/ack pair can live on each side when either PLC may initiate.

Field Manager & Changing Shared Fields

A shared field lives in three places that must stay consistent: the source PLC's share_this_data DB, the combined sync DB (DB200) on every PLC, and the offsets in config.json. The Field Manager automates this: pick a PLC and add (one or several) or remove a field, and it recomputes all offsets and generates the importable TIA .db external-source files (share_this_data.<PLC>.db and synced_data.db) plus the updated config.json. config.json is the single source of truth for field names.

Each preview reports whether the change is hot-safe. Only appending a field to the last PLC in config order is truly hot-safe (nothing shifts in DB200). Adding to any other PLC shifts every downstream PLC's base offset in DB200, and removing repacks the block — both relocate live data, so they need a coordinated rollout.

Pausing the sync

On the Setup page, Pause Sync suspends only the cross-PLC sync write action. The process, web interface and PLC reads keep running; the shared values in DB200 simply freeze at their last-synced state until you resume. The flag is in-memory, so restarting the server always resumes syncing. Use it to close the window where the running server would otherwise write the old DB200 layout onto a PLC that already has the new one.

Recommended workflow for a layout change

  1. In the Field Manager, make the change(s) and click Replace config.json (the running server keeps syncing on its old in-memory config — no disruption yet; the old file is saved as config.json.bak). Download the generated .db files.
  2. On Setup, click Pause Sync — right before touching the PLCs, to keep the disruption window as short as possible.
  3. In TIA, import the new share_this_data.<PLC>.db into each edited PLC and the new synced_data.db into all PLCs.
  4. Restart plc-sync. The new config.json loads and, because a restart clears the pause flag, syncing resumes automatically on the new, consistent layout.

For multiple PLCs in one session: edit a PLC, Replace, then edit the next (the preview always re-reads config.json from disk, so Replace between PLCs is what accumulates the changes). Each share_this_data.<PLC>.db only needs the version from when that PLC was edited; use the last synced_data.db on all PLCs.

Logging

The server creates daily rotated log files:

  • Info Logs: Connection events, synchronization events, etc.
  • Error Logs: Connection issues, read/write errors, or internal server issues.

Troubleshooting

  • Connection Issues: Verify the PLC’s IP, rack, slot, and PUT/GET settings. Ensure that firewall or network rules allow port 102.
  • Data Not Syncing: Check that DBs are not optimized and that absolute addressing is correctly configured.
  • Heartbeat Stagnant: If the heartbeat does not change, confirm that the PLC logic is running and that the correct DB address is read.
  • Reachable but Not Alive: If the server connects but the PLC appears unresponsive, verify the PLC's heartbeat updating logic.

Web-Based GUI Features

  • Configuration Editor: Edit and save the config.json file via the setup page.
  • Combined Data View: View all PLC data combined, refreshed periodically.
  • Individual PLC Control: Inspect and adjust specific PLC variables.
  • Field Manager: Add/remove shared fields and generate importable .db files (see above).
  • Pause/Resume Sync: Suspend the sync writes during a layout change, from the Setup page.
  • Log Viewer: Browse server logs for troubleshooting.
  • Update Credentials: Change the login username and password securely through the setup page.

MQTT Integration

In addition to the HTTPS API, the PLC Sync Server supports MQTT messaging for real-time data exchange. This integration allows the server to both receive and publish MQTT messages, enabling seamless communication with external systems.

MQTT Subscriber Functionality

For each configured MQTT broker, the server establishes a connection and subscribes to topics as defined in the configuration file. Incoming MQTT messages are processed based on their data type (such as BOOL, INT, or REAL) and merged into the overall data set. This mechanism allows external MQTT clients to send updates or commands that can be integrated with the PLC data.

MQTT Publisher Functionality

The server also acts as an MQTT publisher. At every update cycle, it constructs a JSON payload that includes:

  • Server Heartbeat: A counter to verify data freshness and system activity.
  • Combined PLC Data: The aggregated data from all connected PLCs.
  • MQTT Data: Data received via subscribed MQTT topics.

This payload is published to a topic specified in the configuration. External systems can subscribe to these topics to receive real-time updates on the state of the PLC network.

To utilize MQTT features, ensure that your config.json includes valid MQTT configurations with the broker URL, subscription topics, and a publish topic.

Configuration File (config.json) Syntax and Data Types

The config.json file (or its commented equivalent) is the central location for configuring the server, PLC connections, API endpoints, MQTT settings, and operational parameters.

General Structure

The file typically includes sections such as:

  • httpPort: The port on which the HTTPS server will listen.
  • operationInterval: The interval (in milliseconds) between successive data update cycles.
  • maxReconnectAttempts and reconnectInterval: Parameters that control the reconnection behavior for PLC connections.

PLC Configurations

The plcs array contains one or more PLC objects. Each PLC object includes:

  • name: A unique identifier for the PLC.
  • ip: The IP address of the PLC.
  • rack and slot: Siemens S7 connection parameters.
  • dbNumber: The Data Block number used for normal PLC communication.
  • variables: An object defining the variables used by the PLC. Each variable is defined by a descriptor that includes:
    • type: Data type, such as BOOL, INT, or REAL. These types determine:
      • BOOL: Typically represents a single bit within a byte. Mapped as DBx,Xbyte.bit.
      • INT: Occupies 2 bytes, used for integer values.
      • REAL: Occupies 4 bytes, used for floating point numbers.
    • offset: For INT and REAL, this specifies the starting byte offset within the DB.
    • byte and bit: For BOOL types, these define the exact byte and bit where the value is stored.
  • syncDbOffset: Calculated automatically to assign a unique memory region for data synchronization.

MQTT Configurations

The mqttConfigs array holds configurations for one or more MQTT brokers. Each configuration object typically includes:

  • brokerUrl: The URL of the MQTT broker.
  • subscribeTopics: An object where each key is a topic to subscribe to. Each topic maps to an object containing:
    • dataType: The expected data type for the incoming message (e.g. BOOL, INT, REAL). This instructs the server how to convert the incoming string message.
    • syncVariable: The name of the variable that will be updated with the MQTT message’s value.
  • publishTopic: The topic to which the combined PLC data payload (including heartbeat and MQTT data) will be published.

By clearly defining data types in both PLC and MQTT configurations, the server can accurately map and convert data between the raw PLC memory addresses and the external JSON messages used for monitoring and control.

Example config.json File

Below is an example of a config.json file. This file sets up the server’s listening port, operational parameters, and defines configurations for two PLCs as well as one MQTT broker. You can modify this file to match your specific PLC network and MQTT broker settings. 

{
  "httpPort": 8443,
  "operationInterval": 5000,
  "maxReconnectAttempts": 5,
  "reconnectInterval": 1000,
  "syncDbNumber": 10,
  "plcs": [
    {
      "name": "PLC_MAIN",
      "ip": "192.168.1.10",
      "rack": 0,
      "slot": 1,
      "dbNumber": 1,
      "variables": {
        "machine_running": {
          "type": "BOOL",
          "byte": 0,
          "bit": 0
        },
        "current_speed": {
          "type": "INT",
          "offset": 2
        },
        "temperature": {
          "type": "REAL",
          "offset": 4
        }
      }
    },
    {
      "name": "PLC_BACKUP",
      "ip": "192.168.1.11",
      "rack": 0,
      "slot": 1,
      "dbNumber": 2,
      "variables": {
        "machine_running": {
          "type": "BOOL",
          "byte": 0,
          "bit": 1
        },
        "current_speed": {
          "type": "INT",
          "offset": 2
        }
      }
    }
  ],
  "mqttConfigs": [
    {
      "brokerUrl": "mqtt://broker.example.com",
      "subscribeTopics": {
        "plc/updates": {
          "dataType": "INT",
          "syncVariable": "current_speed"
        },
        "plc/machine": {
          "dataType": "BOOL",
          "syncVariable": "machine_running"
        }
      },
      "publishTopic": "plc/data"
    }
  ]
}

In this example:

  • httpPort specifies the HTTPS port the server will use.
  • operationInterval determines how frequently the server updates PLC data.
  • The plcs array defines two PLCs with unique connection settings and variable mappings. For each variable, the type indicates its data format (BOOL, INT, or REAL) and the accompanying properties (such as byte, bit, or offset) determine its exact location in the PLC’s Data Block.
  • The mqttConfigs array configures an MQTT broker with a URL, topics to subscribe to (each with an expected data type and target sync variable), and a topic to which combined PLC data will be published.
